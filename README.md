Введение
Проект представляет собой стек из трех сервисов: веб‑сервер Nginx для раздачи статических файлов и проксирования, сервер приложений на Node.js, и база данных PostgreSQL. Сервисы запускаются совместно с помощью Docker Compose. Для автоматизации предусмотрен конвейер GitHub Actions, который собирает контейнерные образы под популярные архитектуры, публикует их в GitHub Container Registry и отправляет уведомления в Telegram.

Структура и взаимодействие компонентов
Веб‑сервер отдает статический фронтенд и принимает все входящие HTTP‑запросы на корневом адресе. Запросы к пути /api перенаправляются на сервер приложений по внутренней сети Docker.
Сервер приложений реализует эндпоинты и бизнес‑логику, подключается к базе данных и обрабатывает запросы, пришедшие через веб‑сервер.
База данных хранит данные приложения в выделенном Docker‑томе, чтобы переживать перезапуски контейнеров. Доступ к базе извне по умолчанию не публикуется — с ней работает только сервер приложений.
Требования и подготовка окружения
Необходимо установить Docker и Docker Compose (версии 2 и выше), а также Git для получения исходного кода. Для работы CI понадобятся включенные GitHub Actions, доступ к GitHub Container Registry и два секрета для Telegram‑уведомлений: токен бота и идентификатор чата. Интернет‑доступ обязателен для загрузки базовых образов и публикации артефактов.

Переменные окружения
Конфигурация задается через файл .env.example в корне репозитория. В нем указываются учетные данные и параметры PostgreSQL (имя пользователя, пароль, название базы, порт) и параметры подключения приложения к базе (адрес хоста, порт, имя пользователя, пароль, название базы). Значения должны совпадать между секцией базы и приложения, чтобы соединение устанавливалось корректно.

Пример содержимого .env.example:
PGHOST=db
PGUSER=app
PGPASSWORD=secret
PGDATABASE=appdb
PGPORT=5432
PORT=3000

Развертывание локальной среды
1) Получение исходного кода
Клонировать репозиторий и перейти в каталог проекта:
git clone https://github.com/NikitaShamilian/TestTaskForDevOps
cd TestTaskForDevOps
2) Подготовка конфигурации
Создать файл .env и заполнить его по примеру выше. Убедиться, что имя пользователя, пароль и база в секции приложения совпадают с параметрами базы данных.
3) Сборка и запуск сервисов
Собрать образы и запустить контейнеры в фоне:
docker compose up -d --build
4) Проверка доступности
Открыть в браузере адрес:
http://localhost/
5) Остановка и очистка
Остановить и удалить контейнеры:
docker compose down -v

Непрерывная интеграция и публикация образов
Конвейер GitHub Actions запускается при коммитах и запросах на слияние. Он собирает образы фронтенда и сервера приложений для архитектур amd64 и arm64 и отправляет уведомление в Telegram.

Настройка прав и секретов в GitHub
1) Выдать GITHUB_TOKEN права на запись в пакеты: Settings → Actions → General → Workflow permissions → Read and write.
2) Добавить секреты для уведомлений: Settings → Secrets and variables → Actions.
TG_BOT_TOKEN — токен бота Telegram.
TG_CHAT_ID — идентификатор чата.
CR_PAT — токен classic доступа GitHub (scopes: repo, workflow, write:packages, delete:packages, admin:org).

В репозитории используются три ветки с разным назначением и правилами работы. 
main — основная стабильная ветка. Сюда попадает код, прошедший проверку и готовый к использованию.
testing — ветка для интеграционного тестирования. В неё собирают и сводят изменения перед выпуском в main.
fix/idea — ветка для быстрых правок и экспериментальных улучшений небольшого объёма. Подходит для прототипирования и проверки гипотез.

Заключение
Стек поднимается одной командой, взаимодействие сервисов организовано через внутреннюю сеть, а публикация контейнеров и оповещения автоматизированы.
После завершения конвейера отправляется сообщение в Telegram со статусом, ссылкой на репозиторий, коротким хешем коммита и именем ветки. Для получения уведомлений бот должен иметь доступ к чату; для личных диалогов достаточно начать разговор с ботом командой /start, для групп — добавить бота в чат.
